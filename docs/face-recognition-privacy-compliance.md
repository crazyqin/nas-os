# 人脸识别隐私合规检查清单

**审查日期**: 2026-03-29
**审查部门**: 刑部（法务合规部门）

---

## 一、数据采集告知要求（《个人信息保护法》第17条）

| 检查项 | 状态 | 说明 |
|--------|------|------|
| ✅ 告知内容完整 | 已实现 | `privacy.go` 中 `RequestConsent()` 提供完整告知文本 |
| ✅ 告知存储方式 | 已实现 | 明确说明"人脸数据仅存储在本地NAS设备，不上传云端" |
| ✅ 告知数据用途 | 已实现 | 明确用途为"照片人物自动识别和分类" |
| ✅ 告知数据保留期限 | 已实现 | 可配置 `DataRetentionDays`，默认永久保留 |
| ✅ 告知数据删除权利 | 已实现 | `GetPrivacyPolicy()` 包含数据删除说明 |
| ⚠️ 告知形式要求 | 需补充 | 需确保告知以**显著方式**呈现 |
| ⚠️ 告知更新机制 | 需补充 | 需实现告知内容版本更新时的**重新同意机制** |

---

## 二、用户授权机制（《个人信息保护法》第26条）

| 检查项 | 状态 | 说明 |
|--------|------|------|
| ✅ 单独同意 | 已实现 | `RecordConsent()` 记录用户单独同意 |
| ✅ 同意记录存储 | 已实现 | `ConsentRecord` 包含用户ID、同意时间、版本 |
| ✅ 同意状态检查 | 已实现 | `CheckConsent()` 在人脸操作前检查 |
| ⚠️ 同意撤回机制 | 需补充 | 缺少明确的"撤回同意"接口 |
| ⚠️ 非同意状态处理 | 需明确 | 未同意时应完全禁用人脸功能 |

---

## 三、数据存储安全（《网络安全法》第21条）

| 检查项 | 状态 | 说明 |
|--------|------|------|
| ✅ 本地存储 | 已实现 | 明确声明不上传云端 |
| ✅ 加密存储 | 已实现 | `EnableEncryption` 配置项，支持加密存储 |
| ✅ 加密算法 | 已实现 | AES-256-GCM + PBKDF2 |
| ✅ 文件权限 | 已实现 | 数据目录权限 0700，文件权限 0600 |
| ⚠️ 密钥管理 | 需审查 | 密钥存储路径需确保安全 |
| ⚠️ 备份安全 | 需补充 | 人脸数据备份需加密 |

---

## 四、数据删除权利（《个人信息保护法》第45条）

| 检查项 | 状态 | 说明 |
|--------|------|------|
| ✅ 删除功能 | 已实现 | `DeleteAllData()` 删除用户所有人脸数据 |
| ✅ 删除接口 | 已实现 | `DeleteFace()` 单个人脸、`DeleteCluster()` 整个聚类 |
| ✅ 删除审计 | 已实现 | `FacePrivacyAudit.LogPersonDeletion()` 审计删除操作 |
| ⚠️ 删除确认 | 需补充 | 删除前应二次确认，防误操作 |
| ⚠️ 删除彻底性 | 需审查 | 需确保删除后数据不可恢复 |

---

## 五、合规建议汇总

### P0 - 立即处理
1. 添加"撤回同意"接口
2. 未同意时完全禁用人脸功能
3. 实现告知版本更新重新确认

### P1 - 短期处理
1. 人脸数据文件与用户ID绑定
2. 删除后覆写或销毁加密密钥
3. 导出数据加密提醒

### P2 - 中期处理
1. 路径安全验证
2. 文件权限加固
3. 审计覆盖完整性审查

---

**结论**: 项目在人脸识别隐私合规方面已具备基础框架，需要补充同意撤回机制和告知更新机制。

**刑部审查完毕**
