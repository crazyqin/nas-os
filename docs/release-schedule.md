# NAS-OS 版本发布计划

## 发布节奏

**参考**: TrueNAS 26年度发布模式

### 版本号规则

采用**年.月**版本号格式：
- **主版本** (X.0): 年度大版本，重大架构升级
- **次版本** (X.Y): 季度版本，新功能发布
- **修订版** (X.Y.Z): 月度版本，Bug修复和安全更新

示例：
- `v26.1` - 2026年Q1版本
- `v26.2` - 2026年Q2版本
- `v26.1.1` - v26.1的第一个修订版

### 发布周期

| 版本类型 | 周期 | 内容 |
|----------|------|------|
| **主版本** | 年度 | 重大功能、架构升级 |
| **次版本** | 季度 | 新功能、性能优化 |
| **修订版** | 月度 | Bug修复、安全更新 |

## 发布流程

### 1. 开发阶段 (Alpha)

- 功能开发和单元测试
- 代码审查和静态分析
- 每周构建和内部测试

### 2. 测试阶段 (Beta)

- 集成测试和性能测试
- 安全审计
- 用户Beta测试

### 3. 发布阶段 (RC → GA)

- 候选版本(RC)发布
- 最终测试和文档更新
- 正式版本(GA)发布

### 4. 维护阶段

- Bug修复和安全更新
- 修订版本发布
- 用户支持

### LTS版本

每年选择一个版本作为长期支持(LTS)版本：
- 支持周期延长至36个月
- 优先安全更新
- 企业级支持

## 发布节奏调整说明

### 为什么采用年度发布模式？

**参考TrueNAS 26成功经验**:

1. **更可预测的更新周期**
   - 用户可以规划升级时间
   - 企业用户有稳定的维护周期

2. **更高的发布质量**
   - 充分的测试时间
   - 减少紧急修复

3. **更好的资源规划**
   - 开发团队有明确的目标
   - 营销团队有充足准备时间

4. **降低用户升级负担**
   - 减少频繁升级
   - 大版本升级更有价值

### 与TrueNAS对比

| 方面 | TrueNAS 26 | nas-os |
|------|------------|--------|
| 主版本周期 | 年度 | 年度 |
| 次版本周期 | 季度 | 季度 |
| 版本号格式 | 26.1/26.2 | v3.0/v26.1 |
| LTS支持 | 36个月 | 36个月 |
| Beta测试 | 公开Beta | 公开Beta |

## 版本升级策略

### 升级路径

```
v2.283.0 → v26.1 → v26.2 → v3.0.0
```

### 升级建议

1. **主版本升级**: 建议所有用户升级
2. **次版本升级**: 根据功能需求选择
3. **修订版升级**: 建议尽快升级（安全更新）

### 升级兼容性

| 升级类型 | 兼容性 | 回滚 |
|----------|--------|------|
| 主版本 | 可能不兼容 | 需备份 |
| 次版本 | 向后兼容 | 支持 |
| 修订版 | 完全兼容 | 支持 |
