## 🔍 功能详细分析

### 1. AVIF 格式支持

**功能描述**: 为照片管理模块添加 AVIF 图片格式的解码、编码、缩略图生成支持。

| 评估项 | 评分 | 说明 |
|--------|------|------|
| 用户需求 | ⭐⭐⭐⭐ | 用户反馈较多，AVIF 作为新一代图片格式普及度高 |
| 开发成本 | ⭐⭐ | 需要 CGO 依赖 libaom/libavif，跨平台编译复杂 |
| 商业价值 | ⭐⭐⭐ | 提升产品现代化程度，非核心卖点 |
| 风险等级 | ⭐⭐ | 技术成熟，但跨平台编译可能有问题 |

**开发工作量估算**:
- Go AVIF 编解码库调研: 4h
- 核心解码器实现: 8h
- 核心编码器实现: 8h
- 缩略图生成集成: 4h
- WebUI 展示适配: 4h
- 测试覆盖: 4h
- **总计**: 32h (约4人天)

**技术方案**:
```go
// 方案A: 使用纯Go库 (推荐尝试)
import "github.com/AOMediaCodec/libavif-go"

// 方案B: CGO + libavif (备选)
// #cgo LDFLAGS: -lavif
```

**依赖项**:
- libaom / libavif (可选 CGO)
- 测试图片资源

**优先级**: P1 (重要)

### 2. 隐私相册

**功能描述**: 实现加密相册功能，支持密码保护、指纹/面容解锁、隐藏模式。

| 评估项 | 评分 | 说明 |
|--------|------|------|
| 用户需求 | ⭐⭐⭐⭐⭐ | 高频需求，用户隐私保护意识增强 |
| 开发成本 | ⭐⭐⭐ | 需要加密存储、权限隔离、生物识别集成 |
| 商业价值 | ⭐⭐⭐⭐ | 差异化功能，提升用户信任度 |
| 风险等级 | ⭐⭐⭐ | 加密安全性要求高，需要审计 |

**开发工作量估算**:
- 加密存储设计: 8h
- 相册加密/解密服务: 12h
- 权限隔离机制: 8h
- WebUI 加密相册页面: 12h
- 移动端解锁适配: 8h
- 安全审计: 4h
- 测试覆盖: 8h
- **总计**: 60h (约7.5人天)

**技术方案**:
```go
// 加密方案: AES-256-GCM
// 密钥派生: Argon2id
type PrivateAlbum struct {
    ID           string
    Name         string
    EncryptedKey []byte  // 用户密码派生的密钥加密
    Salt         []byte
    Photos       []string
}
```

**依赖项**:
- crypto/aes, crypto/cipher (标准库)
- golang.org/x/crypto/argon2
- 前端 Web Crypto API

**优先级**: P0 (高优)

### 3. 内网穿透优化

**功能描述**: 优化现有 natpierce 模块，提升连接成功率、降低延迟、增加稳定性。

**当前状态**: 
- ✅ 基础框架已实现 (internal/natpierce)
- ✅ P2P 模式和中继模式已支持
- ✅ STUN 协议已集成
- ⚠️ 需要优化连接成功率和稳定性

| 评估项 | 评分 | 说明 |
|--------|------|------|
| 用户需求 | ⭐⭐⭐⭐⭐ | 核心功能，用户反馈频繁 |
| 开发成本 | ⭐⭐⭐⭐ | 基础已有，主要是优化和测试 |
| 商业价值 | ⭐⭐⭐⭐⭐ | 竞品差异化，核心卖点 |
| 风险等级 | ⭐⭐ | 技术成熟，主要是工程问题 |

**开发工作量估算**:
- NAT 类型检测优化: 8h
- 连接重试机制: 4h
- 多 STUN 服务器轮换: 4h
- 心跳保活优化: 4h
- 性能基准测试: 4h
- 稳定性测试: 8h
- 文档完善: 4h
- **总计**: 36h (约4.5人天)

**优化方向**:
1. 智能 NAT 类型检测，选择最优穿透策略
2. 多 STUN 服务器冗余，提高发现成功率
3. 断线重连机制，自动恢复连接
4. 连接质量监控，动态切换 P2P/中继模式

**优先级**: P0 (高优)

### 4. 手机备份

**功能描述**: 实现移动设备照片/视频自动备份功能，支持增量同步、后台备份、智能选择。

| 评估项 | 评分 | 说明 |
|--------|------|------|
| 用户需求 | ⭐⭐⭐⭐⭐ | NAS 核心场景，用户高频需求 |
| 开发成本 | ⭐⭐⭐⭐⭐ | 需要移动端开发、服务端 API、增量同步算法 |
| 商业价值 | ⭐⭐⭐⭐⭐ | 核心功能，用户粘性强 |
| 风险等级 | ⭐⭐⭐⭐ | 移动端开发复杂，需要 iOS/Android 双平台 |

**开发工作量估算**:
- 服务端备份 API: 16h
- 增量同步算法: 12h
- 断点续传机制: 8h
- WebUI 备份管理页面: 8h
- iOS 客户端 (基础): 40h
- Android 客户端 (基础): 40h
- 测试覆盖: 16h
- **总计**: 140h (约17.5人天)

**技术方案**:
```go
// 服务端 API
POST /api/v1/backup/mobile/upload
GET  /api/v1/backup/mobile/status
POST /api/v1/backup/mobile/config

// 增量同步: 基于文件哈希和时间戳
type MobileBackup struct {
    DeviceID    string
    LastSync    time.Time
    FileHashes  map[string]string
}
```

**依赖项**:
- 服务端: 现有 backup 模块扩展
- iOS: Swift + Alamofire
- Android: Kotlin + OkHttp

**优先级**: P1 (重要)
